## ** AWS**

* 一类全新的客户。AWS 起源于与联盟伙伴的数据共享。Colin 接到 Jeff 打来的"那通电话"，Jeff 随即介入。第一届软件开发者大会仅有八人出席。"崇尚创新，力求简化"（Invent and Simplify）这条领导力准则如何让 Amazon 成为 web 服务领域的领跑者。我们如何运用逆向工作法打造出 AWS。*

* * *

* 场景：Colin 的办公室。他的电话响了。来电显示"Jeff Bezos"。*

COLIN：你好，Jeff。

JEFF：你好，Colin。我正在了解我们在 web 服务方面的进展，有人提到了你的名字。你能告诉我，在 web 服务这块，你目前在做些什么吗？如果有的话。

COLIN：当然可以。也许直接给你看最方便。你什么时候方便？

JEFF：现在怎么样？

COLIN* （一边取消接下来的两个会议，一边把笔记本电脑夹在腋下）* ：现在正好。我这就下来。

正如我们已经看到的，在 21 世纪初，从实体媒介向数字媒介的转型对 Amazon 的业务构成了生死攸关的威胁。当时，Amazon 大约 75% 的业务是向客户销售实体图书、CD 和 DVD。我们必须创新，否则就有沦为媒体销售领域一个无足轻重的过气者的风险。此外，尽管 Amazon Prime 取得了巨大成功，但它本质上只是我们既有在线实体媒介零售业务的一个延伸（尽管是非常大的延伸）。

然而，与数字业务或 Prime 不同的是，Amazon Web Services 与核心业务毫无关系。"云计算"这个术语——即通过互联网按需交付诸如计算能力和数据存储之类的 IT 资源，采用按使用量付费的定价方式，而无需购买、拥有和维护物理数据中心与服务器[1](text00025.html#note49a)——在 21 世纪初还未被广泛使用，而且在许多人心目中，Amazon 大概也不在那些有条件提供此类服务的公司名单之列。更何况，它面向的是 Amazon 一类全新的客户：软件开发者。

在本章中，我们不会全面讲述 AWS 的起源和历史。仅这一主题本身就足以写成一整本书。相反，我们将尝试回答以下两个问题，它们可以帮助你把*亚马逊之道*的关键要素融入你自己的组织：
1. *亚马逊之道*中的哪些要素，使 Amazon 得以进入这条完全独立的业务线？
1. 为什么 Amazon 能够远早于其潜在竞争对手——包括那些有庞大既有业务需要保护的老牌公司，以及资本充裕的网络科技公司——就掌握了云计算？

这两个问题的答案，归结起来都在于：单线程团队运用逆向工作法进行不懈迭代，并对客户体验极致专注，从而在云计算这一全新范式下，发现软件开发者的根本需求。

### ** 影响因素**

#### Web 服务的概念验证

那是 2001 年，我（Colin）负责一项名为 Amazon Associates 的联盟业务。这个项目允许第三方（通常被称为联盟伙伴）在自己的网站上放置指向 Amazon 商品的链接。举例来说，正如我们之前提到的，一个关于登山的网站可能会列出一份精选的推荐登山图书清单，并附上指向 Amazon 的链接。当访客点击联盟网站上的某个链接时，就会被带到 Amazon 网站上的图书详情页。如果访客在 Amazon 购买了某件商品，联盟网站的所有者就能赚取一笔推荐佣金。在此之前，联盟伙伴可以选择他们想要展示哪些 Amazon 商品，并通过为数不多的几个参数来决定这些商品在其网站上的展示方式，类似于今天广告服务器的工作方式。这个项目大受欢迎。在我参与该项目的四年间，联盟伙伴的数量从 3 万增长到大约 100 万。但是，正如 Jeff 常说的那样，客户是"神圣地不知足"的，而且"昨天的'惊艳'很快就变成今天的'平常'"。[2](text00025.html#note50a)我们根本没有足够的设计资源，去把 Amazon 商品无缝地整合进我们每一个联盟伙伴的网站中。

正是在那时，我们退后一步，设身处地站在联盟伙伴的角度，从他们的视角来看待这个问题。我们此前一直基于一个正确的假设在运作，即这个项目对联盟伙伴的最大吸引力在于 Amazon 商品本身；但在这么做的同时，我们却忽视了他们对于展示外观和风格拥有选择权的渴望——比如字体大小、配色方案或图片尺寸。原来他们并不愿意将就于"现成最好的"Amazon 格式。

于是，在 2002 年 3 月，我们决定冒一次险，推出一项实验性功能，改变了我们与联盟伙伴共享信息的方式。联盟伙伴可以选择接收一种名为 XML 的文本格式的商品数据，而不是接收一个已经完全成形的商品展示模块。随后，联盟伙伴就能拿这些 XML 商品数据，编写自己的软件代码，按照他们自己的设计标准把商品整合进网站。我们的目标是让自己退出设计这一环，好让他们能够创新，而不必被我们拖后腿。

这项功能在两个方面既新颖又冒险。首先，自 Associates 项目启动以来，我们的核心客户一直是网站所有者。要成为一个联盟伙伴，你不必是软件开发者，甚至不必能够理解或编写基本的计算机代码。这意味着我们必须把一切都做得非常简单。我们会替联盟伙伴生成代码。只需点几下鼠标，就能上线运行。完全不需要编写软件程序。

而这项新功能则不同。它面向的是技术型受众，即那些团队中拥有软件开发者、懂得如何编写代码把商品数据 XML 转换成在自家网站上美观呈现效果的联盟伙伴。我们必须创建一些新的东西，比如用户手册、技术规范和示例代码，把它们全部整合进一个软件开发工具包（SDK）里，向他们展示这套系统是如何运作的。我们还创建了一个讨论板，开发者可以在上面发布他们使用这些服务的心得细节，并相互提问。我们其实并不知道联盟伙伴会对这个更复杂、但也更强大、更灵活的功能作何反应。

这项功能新颖乃至颇具争议的第二个方面在于，它远不只是一个让网站所有者自行设计指向 Amazon 商品链接的工具。顾名思义，商品数据 XML 服务包含了关于 Amazon 商品的丰富信息，比如"购买此商品的客户还购买了 X"。我们煞费苦心地建立了拥有数千万种商品的商品目录，其中还包含了关于消费者对这些商品行为的宝贵数据，公司里许多人都把这份目录视为一项不容外泄的竞争资产。然而，在 Associates 团队看来，让成千上万的开发者在这些数据之上构建商业解决方案，其带来的好处超过了潜在风险。为了降低我们的风险，我们确实对联盟伙伴能用这些数据做什么设置了一些限制。他们只能将其用于销售 Amazon 商品，而且不能永久存储这些数据。除此之外，尽管我们很期待看到联盟伙伴会以怎样富有创意的方式使用这些数据，但再多的分析也无法预测他们究竟会做出什么。

这里又是一个单线程负责人和团队发挥作用的例子。我对联盟业务的财务表现和整体健康状况负责。我们的团队几乎拥有推出这项功能所需的全部资源：我们有软件工程师和产品经理来构建功能；我们有自己的客服代表，他们配备了专门的知识和工具，能够应对联盟伙伴的问询。我们很了解自己的客户，坚信这个实验值得一做，并且愿意在尝试新事物时被人误解。我们还准备了一套回退方案，以应对实验万一失败的小概率情形。

我们决定推出这项功能，看看会发生什么。我们没有发布任何新闻稿，也没有做任何大张旗鼓的公开宣布。我们只是给联盟伙伴发了一封电子邮件，解释这项新功能及其潜在好处，并把他们引导到我们为他们创建的 SDK。我们讲得很清楚，这项功能并不适合所有人。他们得写一些代码，才能让它为自己所用。

每当我们给联盟伙伴发出这样一封邮件时，我都习惯监测他们的反应。我会查看一个仪表盘，上面显示诸如有多少人读了邮件、有多少人点击了邮件里的链接、以及这封邮件带来了多少增量推荐佣金等信息。我还会向客服团队了解他们可能收集到的、关于与联盟伙伴接触的任何轶事性数据，并且我也会去读讨论板上的评论和提问。我得承认，在按下那个众所周知的发送按钮之后，就这封特定的邮件而言，我心里有点忐忑——可能出现的结果，从大有希望到需要花上很长一天来应对坏消息，什么情况都有可能。这份忐忑很快就变成了兴奋。就在发布这项功能后的几个小时内，我就知道我们抓住了一件大事，而且我们的实验将远远超出预期。在讨论板上，联盟伙伴纷纷贴出他们用这项新服务创建的网页链接，还附上他们所用的代码。他们为自己的创造感到自豪，想要与他人分享。他们热情地回答其他联盟伙伴提出的问题，甚至比我们还先一步作答。他们还立即开始提出各种能让这项服务变得更好的新功能建议。

在接下来的几天里，我整理出一份清单，列出那些以令人惊喜和富有创新的方式使用这项服务的网站。有一位开发者创建了一个令人上瘾的游戏，访客要与时钟赛跑、彼此竞速，看谁能根据屏幕上闪现的封面图，最快说出作者、艺术家或电影的名字。另一位开发者让人们能够轻松创建一个网页，把自己个人媒体收藏的虚拟书架陈列出来。最后，还有两个例子，本质上试图重新打造 Amazon 的购物体验，但用的是与 Amazon 网站上完全不同的用户界面。第一个案例我们私下称之为 Amazon Lite。它是一个简洁、精简、以文字为主的网站版本。不花哨，但在小屏幕和功能手机上运行得特别好。（第一代 iPhone 当时还要再过五年才会问世。）第二个案例我们称之为 Amazon Graph，它看起来完全不像一个网站。它是一个应用，展示的是一张由节点和连接节点的线条构成的网络图。每个节点代表一件单独的商品，而这些线条则根据我们的相似性数据指向其他商品。它对我们商品目录的呈现方式令人着迷。这类 Web 应用，在我们发布这项功能之前根本不可能被创造出来。

然后，我就接到了本章开头提到的"那通电话"，Jeff 把我叫到他的办公室。我抓起笔记本电脑，匆匆走出我那间办公室——它坐落在一栋建于 20 世纪 30 年代的装饰艺术风格（Art Deco）建筑里，这栋楼曾经是海洋医院署（Marine Hospital Service）的一部分——下了一段楼梯，走进 Jeff 的办公室。我们围坐在他门边那张用门板做的办公桌旁的会议桌前，我简要地向他讲解了这项新功能。我告诉他，最有意思的洞见并不在于 Amazon 用 web 服务做了什么，而在于*我们的联盟伙伴*用它们做了什么。我接着给他看了一些更有意思的网站和应用，包括上面提到的其中几个，还附上了显示它们所带来流量和销售额的数据。

我告诉 Jeff，自发布以来，软件开发者每天都在用我们做梦也想不到的方式，创造出运用这些功能的应用。

在我们完成这次线上巡览之后，Jeff 评论说，单单一项功能就带来如此的采用率和创新水平，实属罕见，我们需要在这个领域加倍投入。我回答说，我们正在设法在三个月后的 7 月，把它推向更广泛的受众，并配上远为丰富的功能集。从那天起，Jeff 就成了一个热忱的支持者。

看到我们所做之事潜力的，并不只有 Jeff 一人。在 Amazon，想为你的项目多要几个软件工程师，就好比去自动售货机里翻找别人落下的零钱。这种事几乎从不发生。但当我去找我的经理 Neil Roseman，看看有没有人手可用时，他立刻回复我，说有一个由 Rob Frederick 带领的小组刚刚完成了一个名为 Amazon Anywhere 的项目，该项目利用 XML 让 Amazon 的商务功能得以在移动设备上实现。Rob 和他的团队和我们一样，对 web 服务充满热情，于是热忱地加入了这项工作。Sarah Spillman 领导产品管理团队。我们还得到了 CIO Rick Dalzell 以及资深技术专家、副总裁 Al Vermeulen 的大力支持，Al 后来在设计和构建 AWS 的许多关键组件方面发挥了举足轻重的作用。Rob 和 Sarah 撸起袖子，带领技术团队和业务团队投入到推出下一个功能完备版本的竞赛中。Al 和 Rick 则帮助把消息不仅在 Amazon 内部传播开来，还传到了软件行业中几位颇有影响力的早期采用者那里。

接下来的三个月一晃而过。我们感觉自己就像开拓者，正在为我们的新客户——软件开发者社区——创造某种真正非凡的东西。

由于我们在为软件开发者创建项目方面经验不多，我们便向这项服务的重度用户寻求面对面的反馈。我们决定在西雅图总部举办一场 Amazon 软件开发者大会。第一届总共只吸引了八个人。其中两人是我们从欧洲飞过来的。就在大会开始前一周，我才发现其中一位欧洲与会者是个青少年。我不得不去问我们的法务部门这样行不行——所幸我们不需要征得他父母的同意，他得以来参加我们的大会。

我们把各项后勤事务安排妥当，设置了满满一天的会议环节。来自 O'Reilly Media 的 Tim O'Reilly 和 Rael Dornfest 也到场了，他们两人都是 web 服务运动的热忱支持者，教会了我们许多关于这个新领域的知识。另一位与会者是一名恰好住在西雅图的热心客户。他的名字叫 Jeff Barr。他评论道：

与会者的人数比 Amazon 员工还要少。我们坐着聆听讲者谈论他们如何在既有成功的基础上再接再厉，并随着时间推移扩展他们的 web 服务产品。有一位讲者（可能是 Colin Bryar，但我不太确定）展望未来，说他们今后会在公司里四处寻找其他可以对外开放的服务。

对我来说，这就是那个众所周知的灵光乍现的时刻！很显然，他们思考的是开发者、平台和 API，而我想成为其中的一员。[3](text00025.html#note51a)

几周之后，Jeff Barr 加入了 Amazon，至今仍在公司任职，担任 AWS 的副总裁兼首席布道师。

对于在场的人来说，当天最后一个环节大概是最令人难忘的——与 Jeff Bezos 的问答。不用说，那八位与会者中有许多人，既为能有这样一次与 Jeff 亲密交流的机会而激动惊喜，也为他对这项服务了解之深而感到惊讶。在他 2006 年的股东信中，Jeff 清楚地阐明了他的理由：

和任何公司一样，我们的企业文化不仅由我们的意图塑造，也是我们历史的产物。对 Amazon 来说，这段历史相当新鲜，而幸运的是，它包含了好几个由小小种子长成参天大树的例子。我们公司里有很多人，都亲眼见证过多个 1000 万美元的种子成长为价值十亿美元的业务。在我看来，正是这种亲身经历，以及围绕这些成功而生长起来的文化，很大程度上解释了我们为什么能够白手起家创办新业务。这种文化要求这些新业务必须具有高潜力，必须富有创新性和差异化，但它并不要求它们在诞生之日就规模庞大。

我记得 1996 年当我们的图书销售额突破 1000 万美元时，我们是多么激动。要激动并不难——我们从零增长到了 1000 万美元。而今天，当 Amazon 内部一项新业务增长到 1000 万美元时，整个公司正从 100 亿美元增长到 100.1 亿美元。对于那些执掌我们既有的十亿美元级业务的高管来说，对此嗤之以鼻是件很容易的事。但他们没有。他们关注着这些新兴业务的增长率，并发来祝贺的邮件。这相当酷，我们也为它是我们文化的一部分而自豪。[4](text00025.html#note52a)

我在本章开头提出了一个问题：Amazon 究竟是如何第一个进入云计算领域，并成为最大的 web 服务提供商的？Jeff 在他的信中给出了答案：这是因为 Amazon 的创新精神，加上长期思维所带来的耐心。即便在这项业务还全新且非常弱小的时候，我们就意识到它具有高潜力，意识到这是一个我们能够创新和差异化的领域，而且我们有耐心坚持下去。

2002 年 7 月，我们推出了第一个版本的 Amazon Web Services。如果说几个月前我们发给联盟伙伴的商品数据 XML 是测试版（beta），那么 AWS 就是 1.0 版。它包含了一些搜索和购物功能以及一整套软件开发工具包，而且面向任何人开放，不只是联盟伙伴。此外，它当时仍然是免费的。对于这一次，我们确实发布了新闻稿，Jeff 在其中说：

我们正为开发者铺开一块欢迎的门垫——对我们来说，这是一个重要的开端和新的方向……开发者现在可以把 Amazon.com 的内容和功能直接整合进他们自己的网站。我们迫不及待想看看他们将如何给我们带来惊喜。[5](text00025.html#note53a)

在此之前，Amazon 有两类客户——买家和卖家。如今我们有了一类新客户——软件开发者。

发布之后，当我们监测反响时，又遇到了一个惊喜。我们最大的一些客户既不是联盟伙伴，也根本不是任何外部人士。他们是 Amazon 的软件工程师。他们发现，比起自己此前一直用来构建 amazon.com 的一些现有内部软件工具，Amazon Web Services 更好用。到了这个地步，几乎毫无疑问，web 服务将成为一种构建事物的全新方式。我们只是还不知道它究竟能有多大，也不知道开发者会以多快的速度采用它。不到一年，我们就有了一个相当清楚的概念——超过 2.5 万名开发者注册了这个项目[6](text00025.html#note54a)，并不断用他们构建出的东西给我们带来惊喜。

尽管这个项目当时叫作 Amazon Web Services，但它与今天的 AWS 几乎毫无相似之处。事实上，我们在 2002 年推出的这项服务后来被更名为 Amazon Product API，它有一个重大局限——它必须被用于营销 Amazon 商品，因此完全聚焦于改善 Amazon 的零售生态系统。

还有其他一些项目也帮助我们意识到了 web 服务能有多大。同样是在 2001 年，我们启动了一个名为"活页夹"（3-Ring Binder）的项目，它试图创建并记录一套 API，让合作伙伴能够快速地把他们的商品选品添加到 Amazon 网站上，并创建由 Amazon 技术驱动、但使用他们自己的 URL、由他们自己掌控的网站。这个项目最终让我们得以为 Target 等零售商合作伙伴创建网站。此外，我们还开发了一个名为 Seller Central 的项目，它提供了 web 服务，供 Amazon 的第三方卖家用来管理自己的业务。Amazon Associates Product API、Amazon Anywhere、活页夹和 Seller Central，全都印证了我们的一个假设：软件构建的方式正在发生一场天翻地覆的巨变。

2003 年夏天，就在 Web Services 开始腾飞之际，我丢了工作。或者更确切地说，我换了一份工作。正如我在引言中提到的，Jeff 问我是否愿意成为他的技术顾问——他的影子——这是一个我无法拒绝的邀约。在此之前的 18 个月里，Andy Jassy 一直担任 Jeff 的技术顾问，此时他已准备好迎接新的角色。他几乎可以在公司里担任任何职务，包括其最大业务中任何一个的领导职位。对我们来说幸运的是，他决定组建并领导一个新团队，在我们的实验基础上继续发展。Andy 和他的团队构想并创造出一套远为强大的产品，它们将开启云计算的时代，并成长为今天 AWS 这般巨大的成功。虽然 AWS 的爆发式增长及其精密的产品套件全都发生在我转去做其他项目之后，但它仍是*亚马逊之道*的一个生动例证。

值得一提的是，我们当时是有竞争对手的。当时还有其他几家公司也在提供基于 web 服务的开发者项目。和 Amazon Product API 一样，他们的项目意在增强自家的生态系统。举例来说，eBay 的开发者 API 为开发者提供了构建应用的工具，用以在 eBay 上买卖商品。Google 有一个搜索 API，它与 Amazon Product API 在同一周推出。Amazon 还有第二个 web 服务项目，Marketplace 卖家可以用它来管理自己在 Amazon 上销售的商品。这些项目在开发者社区里引起了相当大的反响。

所有这些项目的一个共同点是，它们的最终目标都是让各自的合作方构建出新软件，从而以某种方式给自家的核心业务带来好处——比如更多的 Amazon 联盟销售、更多的 eBay 交易、更多的 Google 搜索，以及更多的 Amazon Marketplace 卖家交易。我们所有人，这些公司的领导者和开发者，看的都是相似的数据和趋势。我们在开发者大会上碰面，一起参加专家小组讨论，还共享着那些使用我们各自开发者项目的客户。我们全都在同一锅"原始汤"里游泳。然而，是 Amazon 迈出了 web 服务的第一步，并说："我们何不构建一套工具，让任何开发者都能用它来构建他们想要的任何东西，哪怕这与我们的核心业务毫无关系？"如前所述，这在很大程度上是因为 Amazon 对创新的专注。"崇尚创新，力求简化"这条领导力准则中有一部分写道："在我们做新事物的时候，我们接受自己可能会在很长一段时间里被人误解。"尽管有怀疑者说 Amazon 不属于这个领域，但我们已经亲身感受到了开发者社区的热情，于是我们对这份热情加倍下注。

还有另外两个因素，影响了我们押注 web 服务这一决定。

#### 基本单元早已为人所知，只是尚未以 web 服务的形式对外开放

几十年来，那些根基稳固的硬件和软件公司，一直在针对构建商业软件所固有的一组众所周知的问题，构建和销售功能强大的解决方案——存储（用于保存和检索数据的数据库）、消息队列和通知（后两者是计算机进程用来相互通信的不同方法）。如果一个软件开发者需要实现这些构件中的某一个，他就得购买一份软件许可证，这通常会产生一笔不菲的一次性费用，外加在产品使用期限内每年都要缴纳的维护费。此外，他要么得购买硬件并在自己的数据中心里运行它，要么就得花钱请合作伙伴来做这件事。

我们不必去发明这些构件——或者说这些"基本单元"（primitives），人们是这么称呼它们的——我们只需想办法把它们作为一种 web 服务在云端提供出来。举例来说，如果你想使用 Amazon 的 S3 存储服务，你所需要做的就是注册一个免费账户，并提供一张信用卡。写上几行代码来设置好你自己的存储区域（这一步称为"预置"）之后，你就可以开始存储和检索数据了。然后你只需为你实际使用的部分付费，这意味着没有耗时的供应商选型流程，也没有价格谈判（许多企业软件许可证的标价，不过是谈判的起点而已）。而且你不必去筹备计算机和数据中心来运行你的新数据库。云服务提供商——在这个例子里就是 Amazon——会把这一切都打理好。

即便在我转去做 Jeff 影子的时候，我也一直关注着这些进展。我换工作后不久，Jeff 和我参加了一场 O'Reilly 新兴技术大会，我们去听了一个专家小组讨论，主讲人是 Stewart Butterfield，他是热门照片分享网站 Flickr 的联合创始人，后来又创办了 Slack。有人请 Stewart 描述一下 Flickr 典型的一天是怎样的。他的回答出人意料。他说，一天中大约有一半时间，大概和台下许多人的情况差不多——手忙脚乱地让他们的技术平台比业务的快速增长快上那么一步。他们忙于扩展自己的数据库、web 服务器、软件和硬件。Stewart 说，他们花在创新那些 Flickr 独有之物上的时间，并没有他希望的那么多。

会后，Jeff 和我就 Stewart 的评论简短地聊了几句。我们俩都注意到了同一件事——一种 Amazon 后来称之为"无差异化的繁重工作"（undifferentiated heavy lifting）的现象，也就是那些我们可以替各家公司代劳的任务，从而让它们能够专注于那些使自己与众不同的事情。这是一个机会。

#### 服务器端对我们来说轻而易举，对其他大多数人来说却困难重重

影响我们决定提供一套更广泛服务的另一个因素是，在构建和运营全球最大网站之一的过程中，我们获得了一项只有极少数公司才能与之匹敌的核心能力。我们有能力存储海量数据，对这些数据进行运算，然后快速而可靠地把结果交付给终端用户，无论对方是人还是计算机。

比方说，假设你想构建一项服务，用来存储数百万张照片，供数百万客户搜索和查询。在 2002 年，对 Amazon 来说，这会是一个相当大、但完全可行的项目。事实上，这差不多就是我们"图书内文搜索"（Search Inside the Book）功能的写照。然而，对大多数公司而言，这样一个项目在成本和时间上都是难以承受的。但很明显，会有越来越多的公司开发或获得这些能力，而它们最终将沦为一种无差异化的大路货。

后来的情况正是如此。今天，构建存储和检索数百万张照片的能力，完全可能只是一名修读计算机科学课程的大学生的家庭作业。在早期 AWS 产品的几份逆向工作法文档中，PR/FAQ 都写明，我们希望宿舍里的学生能够用上和任何一位 Amazon 软件工程师同样世界一流的计算基础设施。PR/FAQ 文档中这个强有力的比喻，确实帮助 AWS 产品开发团队把各种想法和构思梳理得清晰明了。

有若干因素影响了 Amazon 决定涉足 web 服务、并最终成就 AWS 的这一决策。我们在 Amazon Product API 和 Amazon Seller API 上已有几个概念验证，证明这个领域值得关注。相比当时使用的传统方法，它就是一种更好的软件构建方式。由于软件构件早已为人所知、只是尚未以 web 服务的形式提供出来，我们对需要做什么有一份相对清晰的路线图。我们也知道，我们的独特能力不会独特太久，这带来了一种紧迫感。（第一个提供一套强健通用型 web 服务的公司，并不保证能笑到最后，但先发优势肯定大有帮助。）

这种紧迫感被写进了 Amazon 的"崇尚行动"（Bias for Action）领导力准则之中。它写道："在商业中，速度至关重要。许多决策和行动都是可逆的，无需进行详尽的研究。我们看重经过权衡的冒险。"像 Andy Jassy 这样的资深领导者，选择白手起家创办一项新业务、而非接手一项既有业务的执掌权，这在 Amazon 并不罕见。正如 Steve Kessel 和 Bill 从 Amazon 当时最大的业务转到最小的业务之一，或者 Colin 和团队冒险发布一项全新但颇具争议的 web 服务一样，都不足为奇。

我们也有几分运气。在 2015 年的股东信中，Jeff 说道："在任何一项事业中，运气都扮演着超乎寻常的重要角色，而我可以向你保证，我们的运气供应一直很充裕。"[7](text00025.html#note55a)我们很幸运，那些云计算出现之前的老牌巨头以及网络科技公司，动员起来并开始提供自己那套云服务所花的时间，比我们预期的要长得多。等到他们意识到云计算潜力的时候，Amazon 已经领先了好几年。

### ** 起步阶段的 AWS**

那么接下来发生了什么呢？基本上，这场竞赛的第一部分，就是围绕逆向工作法的 PR/FAQ 流程进行长达数月的迭代，同时尽我们所能快速地一个接一个地走完抬杆人（Bar Raiser）流程去招聘候选人，以便着手组建团队。一如既往，我们不走捷径。同样值得注意的是，在最初那约莫半打服务中，只有两个取得了压倒性的成功——Amazon S3（Simple Storage Service，简单存储服务）和 Amazon EC2（Elastic Compute Cloud，弹性计算云）。Jeff 和我每两周会和 Andy 以及这些团队的负责人碰一次面，有时更频繁。此外还有一个庞大的团队，负责构建所有这些服务都会用到的基础设施。这套基础设施由计量、计费、报表以及其他共享功能等组件构成。

尽管最初的基本单元路线图相对简单直接，但真正不那么容易的，是想清楚该如何构建它们，才能让它们在一个比我们为 Amazon 零售业务所做规模高出好几个数量级的量级上运行。这里面有许多棘手的技术难题，为了解决它们，团队做了一些真正令人叹为观止的工程工作。完整的讲述超出了本书的范围，但为了让你有个概念，下面描述的是我们讨论和打磨过的一个关键问题。

逆向工作法的全部要义，在于从客户视角出发，遵循一个循序渐进的流程，不懈地质问各种假设，直到你对自己想要构建的东西有了完整的理解。它关乎追寻真相。有时，逆向工作法可以揭示出一些出人意料的真相。有些公司为了赶着把一个项目推向市场，会无视这些真相，继续按照最初的计划埋头去建。他们执着于那份计划所能带来的微薄收益，激励团队去积极地追逐它，结果却在很久以后才意识到，倘若他们当初肯花时间去质问自己的假设，本可以获得大得多的收益。在 PR/FAQ 撰写阶段改变航向的成本，要远低于你已经发布产品、并有一门运营中的业务需要打理之后再去改。逆向工作法往往能让你免于陷入在产品发布后再做重大方向调整这种代价高昂的境地。有一个能够说明这一点的例子，就是 S3 上出现过的一个问题。

在 FAQ 中有一个简单的问题，大意是："S3 要花多少钱？"最初几个版本的答案是，S3 将是一种基于平均存储用量的、分档的按月订阅服务，对于小额数据量可能设有一个免费档。客户会根据自己通常需要存储的数据量，来选择一个按月订阅的费率——简单存储服务，配上简单的定价。我们还没有敲定各个档位的确切细节以及每一档的价格，但在逆向工作法早期的迭代中，你并不需要把这些都做出来。工程团队已经准备好继续讨论下一个问题了。

只不过那一天，我们从未讨论到下一个问题。我们一直在讨论这个问题。我们实在不知道 S3 推出后开发者会怎么使用它。他们主要存储的是检索率低的大对象？还是检索率高的小对象？更新与读取相比会有多频繁？有多少客户需要的是简单存储（可以轻松重建，只存于一处，丢了也没什么大不了），又有多少客户需要的是复杂存储（比如银行记录，需存于多处，一旦丢失后果极其严重）？所有这些因素都是未知的，却又可能对我们的成本产生实质性影响。既然我们不知道开发者会如何使用 S3，那么有没有一种办法，能够构建我们的定价，使得无论它被如何使用，我们都能确保它对客户和对 Amazon 而言都是负担得起的？

于是，讨论从分档订阅的定价策略，转向了一种"成本跟随"（cost-following）策略。"成本跟随"意味着你的定价模型主要由你的成本驱动，而这些成本随后被转嫁给你的客户。这正是建筑公司所采用的做法，因为用红杉木给你的客户搭凉亭，成本会比用松木高出许多。如果我们采用成本跟随策略，我们就要牺牲掉订阅定价的简洁性，但我们的客户和 Amazon 双方都会从中受益。有了成本跟随，无论开发者用 S3 做什么，他们都会以一种满足自身需求的方式来使用它，并且他们会努力把自己的成本降到最低，因此也就把我们的成本降到了最低。不会有人钻系统的空子，我们也不必去估算那个虚构的"平均客户"会如何使用 S3 来定我们的价。[*](text00024.html#foot4a)

对 S3 来说，最重要的成本驱动因素，会是把数据存在磁盘上的成本吗？是移动数据的带宽成本？是交易次数？还是电力？我们最终敲定了存储和带宽。正如我们在 S3 推出后所发现的，我们的预测有那么一点偏差。用 AWS 的 CTO Werner Vogels 的话来说：

早期有一个例子，我们并不知道服务某些使用模式所需的资源是什么，那就是 S3：我们原本假定存储和带宽是我们应该收费的资源；但运行了一段时间之后，我们意识到请求的次数是一项同等重要的资源。如果客户有许多微小的文件，那么即便他们发出了数百万次请求，存储和带宽也算不上多少。我们不得不调整我们的模型，把资源使用的所有维度都纳入考量，这样 AWS 才能成为一门可持续的生意。[8](text00025.html#note56a)

然而，至关重要的是，我们采用成本跟随策略的这一决定，使我们能够相对轻松地纠正错误、调整定价。既然我们已经在 PR/FAQ 流程中确定了这项服务所有可能的成本驱动因素，我们现在就可以调整定价，使其符合那些真正的成本驱动因素。如果我们当初坚持最初那个订阅定价的想法，这次调整的幅度就会大得多，代价也会高得多。

在我们开发这个早期版本 AWS 的那些日子里，Amazon 才刚刚开始使用逆向工作法。许多团队都对这种做事方式之慢感到沮丧。在我们讨论定价的那次 PR/FAQ 会议上，几位软件工程师坐立不安。会后有一位把我拉到一边，说："我们是软件工程师，不是有 MBA 学位的定价专家。我们想写软件，而不是写更多的 Word 文档。"煞费苦心地遵循逆向工作法，意味着那次会议上我们连 PR/FAQ 文档的其余部分都没讨论到。而且这意味着，工程师们现在得在下次会议之前，对这项服务的相对成本做一大堆研究、测试和测算。我请他们信任这个流程，尽管在那个当下它显得很痛苦。

Jeff 坚持要我们遵循这个流程，直到我们揭示出真相、并对我们究竟想要构建什么了然于胸。他说，以我们想要达到的规模量级，除非这项服务在首次发布时就构建得当，否则团队将把所有时间都花在维持系统运转上，而根本无法开发任何新功能。而事实证明，如果你去看看这些服务中任何一个最初的 PR/FAQ，再拿它和发布时的 PR/FAQ 相比较，它们全都朝着好的方向演进了不少。

你无法把时钟拨回去，重放这个实验，看看如果我们在不知道那些用逆向工作法发现的真相的情况下、就迅速构建并发布这些服务，结果会怎样。然而，尽管发布后仍有一些维护问题和宕机，但其性能表现和客户的迅速采用足以说明一切。基于我在 AWS、数字业务和其他服务的十多个不同产品团队中与 Jeff 一起走完逆向工作法流程的经验，我可以有把握地说，我们放慢脚步、多花时间去揭示那些必要真相，最终反而是通往一门庞大而成功的业务的更快路径。结果不言自明。Amazon 拥有规模庞大、切实可行的数字设备和媒体业务。而且，正如引言中提到的，AWS 达到年收入 100 亿美元里程碑的速度，比作为在线零售商的 Amazon 还要快。

这个早期版本 AWS 的推出，是一个特别好的案例研究，展示了亚马逊一些根本性的原则和流程。"崇尚行动"是 Amazon 一条重要的领导力准则，而在 AWS 这件事上，我们确实承受着要抢在竞争对手之前推出这款产品的时间压力。但"崇尚行动"并不能免除逆向工作法那些煞费苦心的环节。我们没有让自己被竞争对手可能会做什么这件事驱使到如此地步，以至于在还没有非常仔细地思考过客户将如何使用我们的产品、又将从中获益之前，就把产品推出去。换句话说，逆向工作法正是让我们能够把"客户至上"这一原则付诸行动的流程。

多亏了 AWS 的 CEO Andy Jassy 及其团队富有远见的工作，今天的 AWS 比我在 21 世纪初参与的那个版本要庞大得多、也大不相同。但这恰恰正是 AWS 之所以如此富有亚马逊之道的原因所在。当初不过是给我们的联盟伙伴发送商品数据 XML 的一个小小实验，如今已成长为 Amazon 业务的主要事业部之一，2019 年带来了 350 亿美元的收入。在认识到这颗小种子有潜力长成 AWS 这棵参天橡树这一点上，Jeff 和 Amazon 的其他人身体力行地践行了亚马逊的"主人翁精神"（Ownership）、"崇尚创新，力求简化"（Invent and Simplify）以及"敢为大事"（Think Big）这几条原则。
